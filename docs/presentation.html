The problem we set out to solve

2010

Superflood — ~2,000 lives lost, 20M affected. Marala Headworks passed ~260,000 cusecs.

2022

Monsoon floods — 33M affected, $30B+ damage. One-third of the country inundated.

Today

Flood intelligence is fragmented: rainfall, river flows, terrain and population sit in separate systems, joined manually under pressure.

Decision-makers need one screen that fuses rainfall, river forecasts, terrain and exposure into a single, defensible risk assessment — for any basin, province or district — in minutes, not days.
Emphasize the manual-fusion pain point — every number in the room today comes from a different office.

Separation of concerns

Measurement (4 deterministic agents) · Judgement (rule-based coordinator) · Language (LLM, wording only). Each layer independently testable and replaceable.

Compute where the data lives

Heavy geoprocessing (zonal statistics, terrain) runs on Google's Earth Engine servers — our machine only orchestrates. Batched calls: 1–2 round-trips per agent.

Two frontends, one brain

Streamlit for internal iteration; Flask + Leaflet WebGIS for operations. Both call the same agent modules — logic exists exactly once.

The architecture question WILL come from the AI researchers — the answer is the measurement/judgement/language separation.

The specialist agents

🌧️ Weather

In: AOI geometry
Out: 24/72 h observed + 72 h forecast rain (AOI mean & max), data latency
Windows anchor to newest satellite image — staleness is measured, never hidden

🌊 River

In: in-AOI stations + weather's rain outlook
Out: discharge per station (+24…+120 h) classified into FFD categories; worst station
Channel-snapping window fixes grid offsets

⛰️ Terrain

In: AOI geometry
Out: DEM, slope, flow-accumulation GeoTIFFs + AOI area
Static — computed once per AOI, cached forever

👥 Exposure

In: AOI geometry
Out: total population + floodplain population (slope<2° proxy)
e.g. Chenab: 69.1M total, 51M floodplain

Coordinator agent resolves the AOI once, runs the four specialists in sequence, hands the weather agent's rain outlook to the river agent as context, logs every hand-off into an auditable conversation, then fuses components into the score and decision — and commissions the report.
Point out the genuine data hand-off (weather→river): agents talk with real numbers, not theatre.

Agents talk — and the conversation is evidence

coordinator → all agents: New assessment for AOI 'chenab'. Resolving geometry and in-AOI FFD stations.

coordinator → terrain: Prepare static terrain layers for Chenab basin.

terrain → coordinator: Layers ready. AOI area 88,788 km²; DEM/slope/flow-accumulation cached.

weather → river: Rain outlook for your discharge context: 10.9 mm fell (72 h basin mean, 32 h latency); 11.5 mm forecast next 72 h.

river → coordinator: Worst station Trimmu: normal (1,417 m³/s peak over 120 h horizon).

coordinator → decision: Components {river: 0, rain_forecast: 0, rain_observed: 0} → score 0/100 (low).

decision → report: Generate the situation report.

Every message is generated by rules from the real numbers — no LLM involved — and is stored in the saved JSON. The full reasoning chain of any warning can be audited months later.

Risk fusion — transparent, reproducible, rule-based

River · weight 0.5

Worst-station FFD category
normal 0 → low 30 → medium 50 → high 70 → very high 90 → exceptional 100

Forecast rain · 0.3

GFS 72 h AOI-mean (mm)
≥25→20 · ≥35→40 · ≥50→60 · ≥70→80 · ≥90→100

Observed rain · 0.2

GPM 72 h AOI-mean (mm)
same calibrated steps

score = Σ weightᵢ · componentᵢ / Σ weightᵢ  ·  failed agents dropped, weights renormalised, "degraded" flagged

< 20 · LOWroutine monitoring
< 40 · MODERATEadvisory to PDMA/districts
< 60 · HIGHflood alert · pre-position teams
≥ 60 · SEVEREflood warning · activate response

Anyone in this room can recompute the score from the report tables with a calculator. That is deliberate.

The decision ladder mirrors NDMA's advisory→emergency escalation, so outputs drop into existing SOPs.

Calibrated on Pakistan's own flood history

~100 mm

2010 superflood — peak 72 h basin-mean rainfall (GPM backtest, Chenab)

~51 mm

2022 floods — clearly separated from the non-flood baseline

~34 mm

2018 ordinary monsoon (control) — correctly stays LOW

  • Rain thresholds were rejected and recalibrated when the first metric (basin-max) failed the 2018 control — the system is tuned to discriminate, not to alarm.
  • Discharge backtest ranks 2010 > 2022 ≈ 2018 correctly, and quantified a real bias: GloFAS runs ~2× low vs FFD-reported flows (2010 Marala: GloFAS ~114k vs ~260k reported cusecs).
  • That bias is printed as a caveat in every report: river categories are conservative until per-station correction lands. We publish our known weaknesses.
Validation artefacts are reproducible scripts in the repo (scripts/backtest_rain.py, backtest_discharge.py) — hydrologists can rerun them today.
Finding and documenting the GloFAS bias is a credibility asset, not a weakness — lead with it before anyone asks.

Cross-validated against an independent commercial provider

The test

Meteoblue "precip today" map for 08 Jul 2026 (23 rainfall bands, 1–90 mm, ~4 km multi-model) compared zonally against our NASA GPM (observed) and NOAA GFS (forecast) for the same day.

The result

Within Pakistan, all three products agree: a light-rain day (≤12 mm) — matching the dashboard's live values. Spatial rank correlation 0.5–0.7: where Meteoblue paints more rain, our data shows more rain.

Apparent differenceVerified explanation
Meteoblue's heavy bands (12–90 mm) missing from our AOIs0% of those bands lie inside Pakistan — they are in India (76–79°E); no Pakistani AOI is affected
Meteoblue peaks sharper than GFS~4 km model resolves local convection that a 25 km global model smooths — a model-resolution property, not an error
Satellite (GPM) totals lowerObserved day was 18/48 images published at check time — the exact 1–3 day latency the UI already displays
Every difference between our data and an independent provider is explainable, quantified, and already disclosed on-screen.
Run 09 July against Meteoblue's 08-Jul precipitation map; the check scripts are reproducible. Key line: differences are explainable and disclosed — that is what data integrity looks like.

Reporting — automatic, local, fully traceable

Generated after every decision

  • Assessment & decision table · component scores · per-station discharge in m³/s and cusecs · rainfall tables with data latency · exposure · explicit caveats
  • Saved automatically to data\reports\ as Markdown + JSON, filename = AOI + timestamp — nothing overwritten, history accumulates

Operator workflow

  • 💾 Save report button re-saves the on-screen assessment without re-running agents
  • Report history panel — any past report opens in one click (MD or JSON)
  • The JSON twin contains every agent output and the full conversation — a warning issued today can be reconstructed exactly, months later
Reproducibility standard: same inputs → same score → same decision → same report. An inquiry can replay any assessment.
Show a saved sitrep briefly — the caveats section demonstrates institutional honesty.

Engineered for failure — degraded, never wrong

Graceful degradation

  • Any agent failure → component dropped, weights renormalised, "DEGRADED" stamped on banner & report with the failed agent named
  • Zero usable data → decision = manual_review. The system never guesses.
  • GloFAS not yet published → automatic fallback to yesterday's cycle
  • Satellite latency → measured and displayed, not hidden
  • LLM offline → deterministic template text; assessment unaffected
